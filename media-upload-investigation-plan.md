## 1. 문제 요약

브라우저의 미디어 업로드 기능에서 `mp4` 1개와 `png`, `jpg` 여러 장을 함께 업로드했을 때 상단 진행률이 `20%`에서 멈춘 것처럼 보이는 현상이 발생했다. 추가 요구사항으로, 업로드된 이미지들은 브라우저에서 바로 확인 가능해야 하고, 사용자가 순서를 배치 및 재정렬할 수 있어야 한다.

코드 기준으로 보면, 현재 화면의 상단 진행률 바는 "실제 업로드 바이트 진행률"이 아니라 "워크플로 단계 완료율"이다. 따라서 대용량 파일 업로드 중에는 사용자가 업로드가 진행 중인지, 멈춘 것인지 구분할 수 없다.

## 2. 확인한 사실

### 프런트엔드

- [app.js](C:/Users/shs/audio/app/static/app.js:131)에서 미디어 업로드는 `FormData`에 파일들을 모두 담아 한 번의 `POST /api/projects/{pid}/media` 요청으로 전송한다.
- [app.js](C:/Users/shs/audio/app/static/app.js:135)에서는 업로드 요청이 끝난 뒤에만 `current`를 갱신하고 `renderMedia()`를 호출한다.
- [app.js](C:/Users/shs/audio/app/static/app.js:139)~[app.js](C:/Users/shs/audio/app/static/app.js:150)에서 업로드 완료 후 브라우저에 이미지/영상 썸네일을 렌더링한다.
- [app.js](C:/Users/shs/audio/app/static/app.js:151)~[app.js](C:/Users/shs/audio/app/static/app.js:160)에서 드래그 앤 드롭으로 순서를 바꾸고 `/media/order`로 저장한다.
- 업로드 중간 상태를 표시하는 별도의 UI나 상태값이 없다.
- [app.js](C:/Users/shs/audio/app/static/app.js:281)~[app.js](C:/Users/shs/audio/app/static/app.js:291)에서 상단 진행률 바는 5개 단계 완료 여부만 반영하며, 각 단계는 `20%`씩만 더해진다.
- 즉, 업로드 완료 후에는 목록 확인과 순서 변경이 가능하지만, 업로드가 실제로 진행 중인 동안에는 진행률 바가 바뀌지 않고 결과 목록도 늦게 나타난다.

### 백엔드

- [projects.py](C:/Users/shs/audio/app/routers/projects.py:61)~[projects.py](C:/Users/shs/audio/app/routers/projects.py:84)에서 업로드된 파일은 서버에서 순차적으로 저장된다.
- [projects.py](C:/Users/shs/audio/app/routers/projects.py:80)~[projects.py](C:/Users/shs/audio/app/routers/projects.py:81)의 `shutil.copyfileobj(uf.file, f)`는 파일이 모두 저장될 때까지 응답을 반환하지 않는다.
- 업로드 중간 진행률을 DB에 기록하는 로직이 없다.
- [render.py](C:/Users/shs/audio/app/routers/render.py:45)~[render.py](C:/Users/shs/audio/app/routers/render.py:53)의 상태 API는 `tts`, `render`, `youtube upload` 상태만 반환하고, 미디어 업로드 상태는 제공하지 않는다.

### 간단 재현 결과

테스트 클라이언트로 `a.jpg`, `b.png`, `c.mp4`를 같은 요청으로 업로드했을 때:

- 업로드 API는 `200 OK`로 정상 응답했다.
- `media_order`에는 세 파일이 모두 반영되었다.
- 하지만 `tts_state`, `render_state`, `upload_state` 및 각 `*_progress` 값은 모두 그대로였다.

즉, 현재 구조상 "미디어 업로드 진행 상황"은 어디에도 저장되거나 표시되지 않는다. 반면 업로드가 끝난 뒤에는 목록 렌더링과 순서 재정렬 기능이 이미 존재하므로, 이번 계획에서는 이 기능을 더 명확하고 안정적으로 노출하는 것이 중요하다.

## 3. 원인 분석

### 1순위 원인: 진행률 바의 의미가 사용자 기대와 다름

가장 가능성이 높은 원인은 UI 오해다.

- 사용자는 상단 바를 "현재 업로드 퍼센트"로 해석한다.
- 실제 구현은 "전체 제작 워크플로 완료율"이다.
- 스크립트 단계만 완료된 상태에서 업로드를 시작하면 상단 바는 `20%`에 머문다.
- 업로드가 끝나서 `media_order`가 갱신되기 전까지는 `40%`로 올라가지 않는다.

결과적으로 업로드 중에는 정상 동작이어도 "20%에서 멈춤"처럼 보인다.

### 2순위 원인: `fetch` 기반 업로드라서 브라우저에서 업로드 진행률을 표시하지 못함

현재 업로드는 `fetch`로 처리된다.

- 이 구현은 업로드 시작과 완료는 알 수 있지만, 업로드 중간 바이트 진행률은 기본적으로 표시하지 못한다.
- 대용량 `mp4`와 다수 이미지가 함께 묶이면 요청이 오래 걸릴 수 있는데, 그동안 화면에 의미 있는 변화가 없다.
- 특히 사용자가 같은 화면에서 상단의 `20%`만 계속 보게 되므로 멈춘 것으로 인식하기 쉽다.

### 3순위 원인: 서버가 파일을 순차 저장하고, 완료 후 한 번만 응답함

서버는 파일들을 한 번에 받아 각 파일을 순서대로 디스크에 저장한다.

- 파일 수가 많거나 `mp4`가 크면 응답 시간이 길어진다.
- 업로드 중간에 파일별 진행 상태를 반환하지 않는다.
- 중간 실패나 지연이 생겨도 현재 UI는 어디서 막혔는지 설명하지 못한다.

### 4순위 원인: 상태 모델 자체에 "미디어 업로드" 단계가 없음

현재 DB와 상태 API는 `TTS`, `렌더`, `YouTube 업로드`만 추적한다.

- 미디어 업로드 전용 `state/progress/error` 필드가 없다.
- 폴링을 하더라도 프런트가 보여줄 값이 없다.
- 이 구조에서는 UX 보강 없이 원인 파악이 계속 어려워진다.

## 4. 부가 리스크

아래 항목들은 이번 현상의 직접 원인일 가능성보다, 운영 중 혼선을 키울 수 있는 요소다.

- 허용되지 않은 확장자는 [projects.py](C:/Users/shs/audio/app/routers/projects.py:71)~[projects.py](C:/Users/shs/audio/app/routers/projects.py:73)에서 조용히 `continue` 처리된다.
- 사용자는 "몇 개가 업로드됐고 몇 개가 건너뛰어졌는지" 알 수 없다.
- 업로드 요청 중 버튼 잠금, 중복 요청 방지, 취소, 재시도 UX가 없다.
- 업로드 함수는 `async def`이지만 내부 복사는 블로킹 방식이라, 대용량 업로드가 겹치면 응답성이 더 나빠질 수 있다.

## 5. 결론

현재 증상은 "업로드 기능 자체가 20%에서 멈췄다"기보다, 다음 조합으로 인해 그렇게 보였을 가능성이 가장 높다.

- 상단 바가 실제 업로드 퍼센트가 아니라 워크플로 단계 퍼센트인 점
- 업로드 중간 진행률을 보여주는 UI가 전혀 없는 점
- 대용량/다중 파일 업로드가 끝날 때까지 응답이 오지 않는 점
- 업로드가 끝나기 전까지는 브라우저에서 결과 이미지 목록과 배치 결과를 확인할 수 없는 점

즉, 핵심 문제는 서버 저장 로직 그 자체보다 "업로드 진행상황을 관찰할 수 없는 구조", "오해를 부르는 진행률 표현", 그리고 "업로드된 이미지 확인 및 순서 배치 기능이 계획상 명시되어 있지 않은 점"이다.

## 6. 대응 계획

### Phase 1. 진단 가시화

상태: 완료

목표: 멈춤인지, 정상 진행 중인지 즉시 구분 가능하게 만든다.

- 미디어 업로드 전용 상태를 추가한다.
- `media_upload_state`, `media_upload_progress`, `media_upload_error` 같은 필드를 프로젝트 상태에 포함한다.
- 업로드 시작 시 `running`, 완료 시 `done`, 실패 시 `error`로 기록한다.
- 업로드 요청 단위와 파일 수, 총 크기, 저장 완료 파일 수를 로그에 남긴다.
- 허용되지 않은 파일은 무시하지 말고 응답에 `skipped_files`로 반환한다.

### Phase 2. 프런트엔드 UX 분리

상태: 완료

목표: 상단 워크플로 진행률과 실제 업로드 진행률을 분리한다.

- 상단 바는 계속 "전체 작업 단계"로 유지하되 라벨을 명확히 바꾼다.
- 미디어 업로드 영역 안에 별도 진행 UI를 추가한다.
- 업로드 완료된 파일은 브라우저에서 즉시 썸네일 목록으로 확인 가능하게 유지하고, 업로드 직후 자동 갱신이 되도록 한다.
- 이미지와 영상이 섞여 있어도 각 항목이 무엇인지 쉽게 구분되도록 카드 형태와 라벨을 정리한다.
- 사용자가 업로드 후 순서를 직접 배치할 수 있도록 드래그 앤 드롭 정렬 기능을 명시 기능으로 승격한다.
- 정렬 후 저장 성공 여부를 사용자에게 바로 보여주고, 새로고침 후에도 순서가 유지되는지 보장한다.
- 업로드 중에는 "3/12개 저장 완료", "총 480MB 중 210MB 전송" 같은 문구를 노출한다.
- 업로드 중에는 중복 클릭과 추가 업로드를 잠시 막는다.
- 실패 시 토스트 한 줄이 아니라, 실패 파일과 원인을 리스트로 보여준다.

### Phase 3. 실제 업로드 진행률 구현

상태: 완료

목표: 사용자가 `20%` 대신 실제 전송 진행률을 본다.

권장안:

- 프런트 업로드 호출을 `fetch`에서 업로드 진행률을 받을 수 있는 방식으로 교체한다.
- `XMLHttpRequest`의 `upload.onprogress` 또는 브라우저 지원 범위를 확인한 후 대체 가능한 업로드 API를 사용한다.
- 최소 2개의 지표를 표시한다.
- 브라우저 전송 진행률
- 서버 저장 완료 상태

주의:

- 브라우저 전송률 100%가 서버 처리 완료를 뜻하지 않을 수 있으므로, UI 문구를 분리해야 한다.
- 예시: `전송 완료, 서버 저장 중...`

### Phase 4. 백엔드 처리 구조 개선

상태: 완료

목표: 대용량/다중 파일 업로드 시 지연 원인을 줄이고 상태를 더 정확히 제공한다.

- 파일 저장을 파일 단위 루프로 유지하되, 각 파일 완료 시 상태를 갱신한다.
- 가능하면 총 파일 수와 완료 파일 수 기준의 진행률을 DB에 반영한다.
- 대용량 파일 처리 시 메모리와 응답성을 고려해 청크 단위 복사를 검토한다.
- 필요하면 업로드 후 썸네일/미리보기 생성 같은 후처리는 별도 백그라운드 작업으로 분리한다.

### Phase 5. 검증 및 회귀 방지

상태: 완료

목표: 같은 혼선이 다시 생기지 않게 한다.

- 혼합 파일(`jpg`, `png`, `mp4`) 업로드 통합 테스트를 추가한다.
- 대용량 파일 1개 + 소형 이미지 다수 조합 테스트를 추가한다.
- 허용 확장자, 중복 파일명, 부분 실패 케이스를 검증한다.
- 업로드 후 브라우저에 이미지/영상 목록이 즉시 보이는지 검증한다.
- 드래그 정렬 후 `media_order`가 서버에 저장되고 새로고침 뒤에도 유지되는지 검증한다.
- 수동 QA 체크리스트에 "업로드 중간 상태가 보이는지"를 포함한다.

## 8. 완료 기준

아래 조건을 만족하면 이번 이슈는 해결된 것으로 본다.

- 사용자가 업로드 중 `20%에서 멈췄다`고 오해하지 않는다.
- 업로드 중 화면에서 현재 상태를 문장으로 확인할 수 있다.
- 실패 시 어떤 파일이 왜 실패했는지 확인할 수 있다.
- 혼합 파일 업로드 후 정상 완료 시 브라우저에서 이미지/영상 목록을 즉시 확인할 수 있다.
- 사용자가 브라우저에서 업로드된 이미지들의 순서를 배치 및 재정렬할 수 있고, 그 결과가 저장된다.
- 대용량 파일 업로드 중에도 UI가 아무 반응 없이 멈춘 것처럼 보이지 않는다.
- 위 완료 기준을 기준으로 현재 구현을 마쳤다.
